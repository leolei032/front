# 离线包管理平台（node-service 实现说明）

本文描述 **`node-service` 工程** 中 **Preload / 离线包** 管理平台的职责、模块划分与核心流程，便于与同目录《离线包技术方案》对照阅读。客户端 WebView 拦截与离线 SDK 不在 `node-service`。

---

## 1. 定位与边界

| 范围 | 说明 |
|------|------|
| **本服务负责** | 接入应用管理、离线包（业务维度）与版本生命周期、资源解析与校验、全量 zip 与 `manifest.json` 生成、OCS 上传、增量 patch（bsdiff）生成与记录、按应用聚合的 JSON 配置下发、云控（P001）同步、管理台 API |
| **本服务不负责** | 端上拦截、本地缓存命中、下载队列等与客户端 SDK 强相关的逻辑 |

代码上统一以 **`preload`** 为模块名（接口前缀 `/api/preload`、`/node-api/preload`）。

---

## 2. 技术栈

- **框架**：Egg.js（Controller → Service → MySQL / OCS / 外部 HTTP）
- **存储**：MySQL（业务表）、对象存储 **OCS**（`@heytap/ocs` 上传全量包、patch、JSON 配置）
- **资源处理**：`cheerio`（页面静态资源解析）、`archiver`（zip）、`bsdiff-node`（增量 patch）
- **配置**：`config.preload` 为默认 OCS/CDN 域名等；接入应用可单独配置 `ocs` 覆盖

---

## 3. 数据模型（表）

与实现中常量一致（见 `app/service/preload.ts`）：

| 表名 | 用途 |
|------|------|
| `preload_app_record` | 接入应用：`app_id`、云控 `product_id` / `secret_key` / `module_id`、`check_code`、OCS 覆盖 `ocs` 等 |
| `preload_package_record` | 离线包业务：关联多个 `apps`、版本、基准版本、状态、过期时间等 |
| `preload_version_record` | 某包下具体版本：资源列表、`url`（各 app 的 CDN 地址）、`md5`、`size`、`state` |
| `preload_patch_record` | 增量包：版本对、patch 地址与 `md5` |
| `preload_white_list` | 按 `app_id` 维度的资源解析域名白名单 |

建表可参考《离线包技术方案》4.2.4；`node-service` 的 `README.md` 中含 `preload_white_list` 示例。

## 5. 核心业务流程

### 5.1 接入应用

- 创建应用时调用云控 **创建配置模块**（`code: P001`），校验 `product_id` / `secret_key`（`conf.addConfigModule`）。
- 管理员可维护应用维度的 **OCS 配置**（`updateAppOCSConfig`），未配置则使用 `config.preload` 默认桶与前缀。

### 5.2 离线包与版本

- **离线包**（`preload_package_record`）绑定多个接入 `app_id`，含负责人、协作人、包体大小上限、过期时间等。
- **版本**（`preload_version_record`）按 Semver 规则递增创建（`semver` 的 `inc`）；未发布前可编辑资源列表。
- **资源解析**（`resolveResources`）：对入口 URL 请求 HTML，用 cheerio 收集 `script` / `link[stylesheet]` / `img` 等；仅允许 **应用白名单域名**（`getWhiteListByApps` + `ctx.checkSafeDomain`）。

### 5.3 发布全量包（`releaseVersion`）

1. 校验资源 URL（HEAD）、白名单、与同一应用下其它包 **资源 URL 不重复**。
2. 逐个 URL 下载内容，按 URL 的 MD5 作为文件名落盘，记录 `md5`、`content-type`、过滤后的 `headers`，写入 **`manifest.json`**（结构见业务协议文档）。
3. 用 **archiver** 打 `packages.zip`，校验体积不超过包配置 `max_size`。
4. 按各接入应用 OCS 配置上传至 `{prefix}{packageId}/{version}.zip`，更新版本记录与包记录（生效版本、发布时间等）。
5. 若存在 **基准版本** `base_version`：通过 **messenger** 投递 `generatePatchFiles`，在 Agent 进程异步做 bsdiff；否则直接 `updateConfigByApps` 刷新配置。

### 5.4 增量包（`generatePatchFiles`）

- 在基准版本与当前版本之间，对历史版本全量 zip 下载后，用 **bsdiff** 生成 `{旧版本}_{新版本}.patch`，上传 OCS，并写入 `preload_patch_record`。
- 完成后同样调用 `updateConfigByApps`，保证客户端能拿到最新列表与 patch 信息。

### 5.5 客户端可见配置（`updateConfigByAppId`）

- 按 `app_id` 聚合所有「未删除且未过期」的离线包，组装 `PreloadAppConfig`（含各包状态、版本、全量 `url`/`md5`/`size` 等）。
- 计算 **`JSON.stringify(data.apps)` 的 MD5** 作为 `check_code`；变化时上传 `{prefix}{app_id}.json` 至 OCS。
- **非 prod**：上传成功后调用云控 `publishModuleRules`（P001，`checkCode`）。
- **prod**：先通过 messenger 触发 **CDN 刷新校验**（`checkCDNRefresh`）：轮询 JSON 直到 MD5 一致后再 `publishModuleRules`；失败则 TT 机器人告警（与 `syncAppCloundConfig` 类似逻辑）。

### 5.6 定时任务

- `app/schedule/update_config.ts`：每日触发，对所有接入应用执行 `updateConfigByAppId`，用于配置与过期等场景的兜底刷新。

### 5.7 开放接口（Node API）

- `POST /node-api/preload/automaticCreateVersion`：服务端流水线等通过 **AES 解密** 的 body 创建/更新版本草稿（需配置在 OAuth `routerWhiteList` 且免 CSRF）；内部校验用户权限与资源列表。

## 6. HTTP 接口一览（管理台为主）

以下为 `app/router.ts` 中登记的能力分类，具体参数以 Controller 与 Service 为准。

- **初始化/运维**：`initBaseVersion`、`initPatchFiles`、`initVersionUrls`（GET）
- **应用**：`getAppList`、`getAppBaseList`、`createApp`、`updateApp`、`deleteApp`、`getAppConfig`、`syncAppConfig`、`syncAppCloundConfig`、`updateAppOCSConfig`
- **离线包**：`createPackage`、`getPackageList`、`updatePackage`、`updateBaseVersion`、`deletePackage`
- **版本**：`getVersionList`、`createVersion`、`getVersionDetail`、`updateVersion`、`releaseVersion`、`copyVersion`、`deleteVersion`
- **Patch**：`getPatchList`、`refreshPatchFiles`
- **资源**：`resolveResources`
- **远程包文件**：`listFilesOfPackage`、`deleteFileOfPackage`、`uploadPackageFile`
- **白名单**：`getAppWhiteList`、`addAppHost`、`deleteAppHost`

权限上，部分接口仅 **系统管理员** 可操作；包与版本在 **应用 Owner / 协作人** 范围内可编辑。

## 7. 与总方案文档的对应关系

- **manifest 协议、zip 命名、增量 patch 命名**：与《离线包技术方案》附录一致；实现见 `releaseVersion` / `generatePatchFiles`。
- **云控 P001**：实现于 `app/service/conf.ts`，与方案 4.4.4 一致。
- **资源解析域名白名单**：实现为 `preload_white_list` + `checkSafeDomain`，与方案 4.2.3.1 思路一致（域名列表以配置为准）。

---

## 8. 配置与运维注意

- OCS 的 `accessKeyId`、`secretAccessKey`、`endpoint`、`bucket`、`prefix`、`domain` 在 `config/config.default.ts` 与 `config/config.prod.ts` 中按环境区分；**生产密钥需按公司规范保管，勿提交到公开仓库**。
- 多进程下 **patch 生成**、**CDN 刷新后再同步云控** 依赖 Egg `messenger`/`agent`，部署时需保证多进程与 Agent 正常。
