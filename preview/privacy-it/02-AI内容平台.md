### Q1-1: 为什么选 Next.js 而不是前后端分离？

两个关键需求决定了必须用 SSR 框架：

1. **SEO 需求**：120w+ 多语言落地页的程序化 SEO，纯 SPA 对爬虫不友好
2. **首屏性能**：面向印尼市场，设备和网络条件参差不齐，SSR 让首屏更快

选 Next.js 而不是 Nuxt/Remix：团队 React 技术栈更熟悉；ISR 能力适合"内容更新频率低但页面量巨大"的场景；App Router 的 Server Components 天然减少客户端 JS bundle。

trade-off：API Routes 不适合重计算，复杂业务逻辑应该拆独立服务。

### Q1-2: 你说独立完成全流程，具体怎么理解"独立"？

独立是指技术侧的端到端交付。但不是完全没有协作：产品方向和业务目标跟 leader 对齐；UI 没有专职设计师，参考了 Airbnb/Booking/TripAdvisor 自己做；数据源是已有的业务数据。

### Q1-3: 为什么选 ES 而不是 PostgreSQL 全文搜索或 Pinecone？

| 维度 | PostgreSQL | Pinecone | Elasticsearch 8.x |
|------|-----------|----------|-------------------|
| BM25 全文搜索 | 支持 | 不支持 | 成熟 |
| kNN 向量搜索 | pg_vector 不够成熟 | 强 | 原生支持 |
| 结构化过滤 | 强 | 弱 | 强 |
| 中文分词 | 弱 | 不涉及 | smartcn 开箱即用 |

选 ES 因为一个引擎解决所有需求（BM25 + kNN + 结构化过滤），架构最简单。数据量 2400 条，单分片搞定。

### Q1-4: App Router vs Pages Router？遇到什么坑？

选 App Router 因为：Server Components 默认减少 JS bundle；嵌套布局更优雅；流式渲染配合 Suspense。

**遇到的坑**：
1. **Hydration Mismatch**：严格控制 `'use client'` 范围，数据获取和格式化留在 Server Components
2. **缓存行为不透明**：所有 fetch 显式设置 `cache` 和 `revalidate`，不依赖框架默认值
3. **Client Components 传染性**：`'use client'` 是边界线，import 的模块都降级为客户端代码。用 children/slot 模式嵌入 Server Component：

```tsx
// page.tsx (Server Component)
<ClientWrapper>
  <ServerChild />   {/* 仍然在服务端渲染，JS 不进 bundle */}
</ClientWrapper>
```

### Q1-5: shadcn/ui 的工作原理？和传统组件库的本质区别？

传统组件库是 npm 包（源码在 node_modules），shadcn/ui 是**代码生成器**——把组件源码复制到你的项目里。意味着：完全可控可改、没有版本依赖、按需引入、底层用 Radix UI 无障碍原语。

项目中 shadcn 组件放 `components/ui/`（禁止直接导入），上层封装 `components/common/qpon-ui/` 暴露给业务。

### Q2: AI 对话式搜索是怎么设计的

**回答（PCTCR）**：

**P**：用户想通过自然语言找餐厅，如"雅加达适合约会的日料"，传统搜索无法理解复合意图。

**C**：2400+ 篇文章维度复杂；需要区分探索搜索和精确店名搜索；首字节延迟尽可能短。

**T**：三阶段 RAG 架构，核心思路是"LLM 做意图理解和内容生成，检索交给确定性的 ES 引擎"：

```
用户 query
  -> Stage 1: Gemini 意图提取 + Embedding 并行（~1.5s）
    -> Stage 2: ES 混合检索 kNN + BM25 + function_score（~80ms）
      -> Stage 3: Gemini 流式生成 Markdown 图文回答（SSE 推送）
```

**关键设计决策**：

1. **Stage 1 意图提取**用 `generateObject`（Zod Schema 约束），提取 6 个字段：
   - `intentType`：chitchat / new_search
   - `searchMode`：explore（多店推荐）/ specific（单店详情）
   - `city`、`cuisine`、`isHalal`：3 个硬筛维度
   - `semanticQuery`：3-8 个语义关键词

2. **Stage 1 + Embedding 并行**：意图提取和 Embedding 生成同时发起，节省 ~500ms。Embedding 直接用原始 query（不等 semanticQuery），因为语义足够接近。

3. **Stage 2 混合检索** ES DSL：

```json
{
  "knn": {
    "field": "embedding",
    "query_vector": [0.12, -0.34, ...],
    "k": 16, "num_candidates": 200,
    "filter": { "bool": { "must": [
      { "term": { "level1Id": "IDN.7_1" } },
      { "term": { "cuisine": "日本料理" } }
    ] } }
  },
  "query": {
    "function_score": {
      "query": { "multi_match": {
        "query": "约会 浪漫 日料",
        "fields": ["summary^2", "poiName.text^3", "titleZh", "titleEn", "titleId"]
      } },
      "functions": [{ "field_value_factor": {
        "field": "googleRating", "factor": 0.1, "modifier": "log1p"
      } }],
      "boost_mode": "sum"
    }
  }
}
```

三路分数求和排序：
- **kNN 向量相似度**：语义层面的匹配（"浪漫约会"能匹配到"氛围好的西餐厅"）
- **BM25 关键词匹配**：精确文本命中，字段有优先级——店名命中权重最高（×3），摘要次之（×2），三语标题兜底（×1）
- **googleRating 加分**：Google 评分通过 `log1p` 平滑后加权（`factor: 0.1` 控制影响约 10%），避免高分店完全压过语义相关性

4. **Stage 3 流式生成**：
   - 先通过 SSE 发送文章元数据（图片 URL 映射），编码为 `<!--ARTICLES:json-->` 前缀
   - LLM 流式输出 Markdown，插入 `![店名](IMG:articleId)` 图片标记
   - 前端实时解析 Markdown，替换 IMG 标记为真实图片
   - 单店搜索（searchMode=specific）使用专门的 prompt 聚焦介绍

**R**：首字节约 2s（Gemini API 是主要瓶颈），完整回答 5-8 家餐厅约 8-10s。48 个场景测试通过率 93.8%。

### Q2-1: 为什么分两次 LLM 调用？合成一次不行吗？

分开是为了可控性。合成一次时 LLM 经常：提取的城市 ID 格式不对、检索参数和生成不一致。分开后：Stage 1 用 Schema 约束提取质量稳定；检索是确定性代码；Stage 3 只做一件事。代价是多一次 LLM 调用，但 Stage 1 和 Embedding 并行执行，实际增加的延迟很小。

补充：当前架构是**单轮搜索**，每次搜索独立，不维护多轮对话历史。这样做的好处是实现简单、无需管理上下文窗口，且搜索场景下用户意图通常自包含。

### Q2-2: searchMode 是怎么设计的？

意图提取阶段就区分 `explore`（探索推荐）和 `specific`（搜店名），不同模式走不同 prompt：

- **explore**："雅加达日料推荐" → 推荐 5-8 家，分组对比
- **specific**："401 Thieves" / "莆田餐厅" → 聚焦一家店，深度探店笔记风格

判断逻辑在 LLM 侧：如果用户输入是完整的店名/品牌名（专有名词），选 specific；否则选 explore。

### Q2-3: 降级策略怎么设计的？

三层降级保证有结果返回：
1. **Embedding 失败** → 去掉 kNN，纯 BM25 检索（关键词匹配仍然有效）
2. **Cuisine 结果不足** → 去掉 cuisine 硬筛，用语义匹配补足
3. **库存耗尽**（结果 < 3 条）→ 去掉排除列表，允许重复推荐

每层降级都有日志记录，便于分析数据覆盖缺口。

### Q2-4: 前端流式渲染怎么做？

**整体数据流**：

```
后端 SSE 流 → useChat 自动拼接文本 → useStreamParser 解析/拆块 → 按 block 类型分发渲染
```

**第一层：SSE 接收（useChat）**

用 Vercel AI SDK 的 `useChat` hook，内部自动管理 SSE 连接，把后端推的 chunk **持续拼接**到 `messages` 数组。前端不需要手动操作 `EventSource` 或 `ReadableStream`：

```tsx
const transport = new DefaultChatTransport({ api: '/api/chat-v3', body: { locale } });
const { messages, sendMessage, status } = useChat({ transport });
const isStreaming = status === 'streaming';
```

**第二层：元数据提取（extractArticlesFromText）**

后端在流式输出**最开头**先发一段 HTML 注释，包含文章元数据（图片 URL 映射）：

```
<!--ARTICLES:[{"articleId":"123","imageUrls":["url1","url2"],"poiName":"...","title":"..."}]-->
```

前端用正则 `<!--ARTICLES:([\s\S]*?)-->` 提取 JSON，构建 `imageMap`（articleId → 图片URL数组），同时从文本中删掉注释，剩下纯 Markdown。这样 LLM 只需输出轻量的 `![店名](IMG:articleId)` 占位符，图片 URL 不进 LLM 上下文窗口。

**第三层：流式安全切割（splitWithIncompleteMarker）**

流式传输中文本可能在标记中间被截断（如 `![寿` 或 `[SUGGEST:`），解析器检测末尾是否有**未闭合的标记**：

```typescript
// 检测最后一个 ![ 或 [SUGGEST: 是否已闭合
const lastOpen = Math.max(text.lastIndexOf('!['), text.lastIndexOf('[SUGGEST:'));
// 未闭合 → 截断部分暂存 buffer，只渲染已闭合的 complete 部分
return { complete: text.slice(0, lastOpen), buffer: tail };
```

**第四层：文本 → ContentBlock[]（parseContentBlocks）**

把安全切割后的文本按正则拆分为三种 block：

| 原始标记 | Block 类型 | 渲染方式 |
|---------|-----------|---------|
| 普通 Markdown 文本 | `markdown` | react-markdown + 自定义组件映射（h2/strong/blockquote/ul/li → 设计系统组件） |
| `![店名](IMG:articleId)` | `image` | 从 imageMap 查真实 URL → ImageGrid 组件（1张大图 / 2-3张横排等宽） |
| `[SUGGEST:推荐问题]` | `suggest` | 流结束后渲染为"你可能还想问"可点击按钮列表 |

LLM 回答末尾通过 `---` 分隔出 Tip 卡片（选餐建议），`MarkdownRenderer` 检测分隔线后将内容包裹在暖色圆角卡片中。

**第五层：状态驱动 UI 切换**

```
用户输入 → 骨架屏（waitingForResult = true）
         → 首个 chunk 到达（hasNewResult = true）→ 实时渲染图文流
         → 流结束（isStreaming = false）→ 显示参考来源卡片 + 推荐问题
```

- 参考来源只展示 AI 通过 `IMG:articleId` **实际引用**的文章（未引用的过滤掉）
- 点击图片/参考来源卡片 → BottomSheet 打开文章详情
- 防闪烁：`imageMap` 用 useRef 缓存，输入区用 `React.memo` 隔离 re-render

**附加功能**：
- **语音输入**：集成 Web Speech API（`useSpeechRecognition` hook），支持中/英/印尼语三语语音识别，实时转文字后自动填入搜索框
- **搜索结果分享**：搜索结果可一键分享，后端将完整结果（query + content + articles）存入 Redis（24h TTL），生成分享页 `/{locale}/chat-v3/s/{shareId}`（SSR 直出），支持 Web Share API + 剪贴板兜底

### Q2-5: 如何防止 LLM 幻觉？

三层防护：
1. **Prompt 约束**：明确要求只基于提供的信息回答，不编造
2. **RAG 架构**：LLM 知识边界被限制在检索结果内
3. **前端校验**：参考来源只展示 AI 通过 `IMG:articleId` 实际引用的文章，未引用的不显示

48 场景端到端测试中，0 条幻觉（输出的店名全部来自搜索数据）。

### Q2-7: 准确率怎么评估？

建了两套测试体系：

**Stage 1+2 快速测试**（48 场景，~2s/query）：验证意图提取准确性 + 搜索结果相关性
- 通过率 93.8%（45/48）
- 检查：intentType、city、cuisine、isHalal 提取准确性
- 检查：搜索结果的 cuisine/city 与 intent 一致性
- 检查：搜店名时 top3 是否命中目标

**端到端 E2E 测试**（22 场景，~10s/query）：验证最终输出内容质量
- 通过率 81.8%（18/22）
- 检查：输出关键词命中率、语言正确性、无幻觉、图片数量
- 失败 case 主要是英语 locale 时 LLM 用中文回答（已通过独立语言 prompt 修复）

### Q2-8: 多语言搜索怎么解决的？BM25 只有中文 summary 不是匹配不上吗？

**现状**：summary 只有中文版，英文/印尼语用户搜索时 BM25 在 summary 字段确实打折扣。但有两层补偿：
1. **kNN 向量搜索兜底**：embedding 模型是 `text-multilingual-embedding-002`，天然跨语言，英文 query 的向量和中文 summary 的向量仍然语义接近，所以召回不受影响
2. **Stage 3 生成层补偿**：en-id 和 id-id 的 system prompt 要求 LLM 将中文 summary 翻译后再回答，保证输出语言正确

**待优化**：离线生成三语 summary（`summaryEn`、`summaryId`），写入 ES 后加入 `multi_match` 字段，从根本上解决 BM25 多语言召回问题。成本很低（2400 条，Gemini Flash 翻译一次几分钟），是明确的 TODO。

### Q2-9: 如果数据量从 2000 篇增长到 20 万篇，架构要怎么变？

- ES 分片从 1 → 3-5，加副本
- Embedding 批量处理增加并发
- 意图提取不变（LLM 不依赖数据量）
- 搜索可能需要 pre-filter（先按 city 分桶）
- 考虑 Embedding 缓存（相似 query 复用向量）

### Q3: P95 延迟 146ms、QPS 900+ 是怎么做到的

**四级缓存命中链路**：

```
用户请求
  -> PWA Service Worker Cache (短 TTL, 离线可用)
    -> CDN Edge Cache (就近节点, 中等 TTL)
      -> Next.js ISR Cache (stale-while-revalidate)
        -> Redis (7d TTL, API层兜底)
          -> ES / BigQuery (数据源)
```

绝大多数请求在前两层被拦截，穿透到数据源的极少。

### Q3-1: 缓存一致性怎么保证？

**最终一致性**策略（内容更新频率低）：
- **主动失效**：内容更新时触发 CDN purge API + Redis DEL + `revalidatePath`
- **被动过期**：各级 TTL 自然过期（PWA < CDN < ISR < Redis）
- **兜底机制**：stale-while-revalidate，用户始终有内容看

```typescript
async function invalidateArticle(slug: string) {
  revalidatePath(`/article/${slug}`);  // Next.js 缓存
  await redis.del(`article:detail:${slug}`);  // Redis
  await cdnPurge(`/article/${slug}`);  // CDN（可选）
}
```

### Q3-2: 缓存穿透/击穿/雪崩怎么防？

**穿透**（查不存在的数据）：空值缓存 5 分钟

```typescript
if (!result) {
  await redis.set(key, 'NULL', 'EX', 300);
  return null;
}
```

**击穿**（热点 key 过期瞬间）：ISR 的 stale-while-revalidate 天然避免（过期后仍返回旧缓存），所以未实现互斥锁。

**雪崩**（大量 key 同时过期）：TTL 加随机抖动

```typescript
const ttl = 3600 + Math.floor(Math.random() * 300);
```

### Q3-3: 压测数据

| 场景 | QPS | 成功率 | P50 | P95 | P99 |
|------|-----|--------|-----|-----|-----|
| 首页单压 | **903** | 100% | 22.9ms | 146.4ms | 188.0ms |
| 推荐单压 | **887** | 100% | 19.9ms | 180.0ms | 224.2ms |
| 组合压测 | **780** | 100% | 19.0ms | 183.7ms | 234.1ms |

50 并发、600 秒/场景、10 秒超时。**诚实说明**：GCP 内网压测，不含用户端网络延迟。

### Q3-4: 扩到 5000 怎么做？

- GKE HPA 水平扩展 Pod
- Redis 不是瓶颈（2400 条数据）
- ES 不是瓶颈（单分片 <50ms）
- 真正瓶颈在 LLM 调用：Gemini API 有 QPM 限制，需要请求排队或模型降级

## Part 4: 程序化 SEO（120w+ 页面）

### Q4: 120w+ 多语言落地页怎么实现

## Part 8: 最难解决的问题（高频问法）

> 面试官问法：
> - "这个 AI 项目中遇到最大的技术挑战是什么？"
> - "从 0 到 1 做全栈，最卡你的地方在哪？"
> - "RAG 这条链路调通容易吗？踩了什么坑？"

### 难点 1：流式 Markdown 渲染中的"截断解析"问题

**发现问题**：
LLM 流式输出到前端时，Markdown 文本会在任意位置被截断。一开始用自定义正则解析 Markdown，频繁出现渲染闪烁、图片重复渲染、列表结构错乱等问题。比如 `![寿司店` 这种半截标记被当成普通文本渲染了，下一个 chunk 到达后又重新解析，导致页面跳动。

**分析根因**：
问题不在 Markdown 解析器本身，而在于 **流式传输中没有"安全切割点"的概念**。任何一个 chunk 边界都可能落在 `![...](...)`、`[SUGGEST:...]`、`**粗体**` 这些标记的中间。如果直接把 incomplete 的文本扔给解析器，结果是不可预测的。

**解决过程**：
1. **初版**：每次 chunk 到达就全量重新解析 → 性能差、页面闪烁严重
2. **二版**：尝试增量追加解析 → 半截标记导致解析树结构错误，状态难以恢复
3. **最终方案**：设计 `splitWithIncompleteMarker` 算法
   - 每次 chunk 到达时，检测文本末尾是否有未闭合的标记（`![`、`[SUGGEST:`）
   - 如果有，把未闭合部分暂存到 buffer，只渲染已闭合的 complete 部分
   - 下一个 chunk 到达后，buffer + 新内容合并，再次检测
   - 完整闭合的标记才会被送入 `parseContentBlocks` 做类型分发（markdown / image / suggest）

```
chunk 1: "推荐这家店\n\n!["
         → complete: "推荐这家店\n\n"  → 渲染
         → buffer: "!["               → 暂存

chunk 2: "寿司店](IMG:123)\n\n下一家"
         → 合并: "![寿司店](IMG:123)\n\n下一家"
         → complete: 全部闭合 → 渲染图片 + 文本
         → buffer: ""
```

**同时迁移到 react-markdown**：自定义正则解析器维护成本太高，改用 react-markdown + 自定义组件映射（h2/strong/blockquote/ul/li → 设计系统组件），只在外层做安全切割和 block 类型分发。

**为什么这是最难的**：这不是一个"搜一下就有答案"的问题。流式 Markdown 渲染在大多数 AI 产品中都是痛点，没有成熟的开源方案可以直接用。需要自己设计切割算法，并且处理各种边界 case（嵌套标记、连续图片、标记跨 3 个 chunk 等）。

### 难点 2：RAG 搜索准确率从 ~70% 提升到 93.8% 的调优过程

**发现问题**：
RAG 链路初版跑通后，48 个测试场景只有约 70% 通过率。主要问题：
- 意图提取不准：用户说"雅加达附近的日料"，Gemini 提取出错误的城市 ID
- 搜索结果不相关：用户问"浪漫约会餐厅"，返回了快餐店
- 多语言混乱：英语 locale 下 LLM 用中文回答

**分析根因**：不是单一问题，而是 **三个环节各有短板**，需要逐个击破。

**逐步解决**：

**Stage 1 意图提取优化**：
- 问题：自由文本输出的城市 ID 格式不稳定（有时 "IDN.7_1"，有时 "Jakarta"）
- 解法：改用 `generateObject` + Zod Schema 强约束输出格式，城市 ID 用 enum 限定候选值
- 效果：城市提取准确率 85% → 98%

**Stage 2 搜索排序优化**：
- 问题：纯 BM25 关键词匹配无法理解"浪漫"="氛围好"，纯向量搜索精确店名搜不到
- 解法：混合搜索三路加权（kNN 向量 + BM25 关键词 + googleRating 加分），字段权重反复调试（店名 ×3、摘要 ×2）
- 关键发现：googleRating 加权的 factor 从 1.0 调到 0.1，用 `log1p` 平滑。factor=1.0 时高分店完全压过语义相关性，一家 4.9 分的无关餐厅排在了语义完美匹配的 4.2 分餐厅前面

**Stage 3 多语言输出**：
- 问题：通用 prompt 下 LLM 经常"切换语言"，英文 locale 夹杂中文
- 解法：为 zh-id、en-id、id-id 三个 locale 各写独立 system prompt，明确语言要求并要求翻译中文 summary

**最终效果**：48 场景意图测试通过率 93.8%，22 场景 E2E 测试通过率 81.8%。

**为什么这是难的**：RAG 调优不像传统 bug 有明确的对错，而是一个 **多变量联合调参的过程**——改了搜索权重可能提升了 A 场景但恶化了 B 场景，需要建立系统化的测试集才能避免"按下葫芦浮起瓢"。

### 难点 3：独立全栈从 0 到 1——技术选型决策的压力

**情况**：
这个项目没有技术负责人指导，所有技术决策都是我独立做的。从框架选型（Next.js vs Remix vs 前后端分离）、搜索引擎（ES vs PostgreSQL vs Pinecone）、AI 模型（Gemini vs GPT-4 vs Claude）到部署方案（GKE vs Cloud Run vs Vercel），每一步都需要在 **信息不完整的情况下做判断**。

**最难的一个决策——搜索引擎选型**：
- 初期考虑 Pinecone（纯向量数据库）：语义搜索能力强，但不支持 BM25 全文搜索和结构化过滤
- 考虑 PostgreSQL + pg_vector：全能但向量搜索性能和生态不成熟
- 最终选 ES 8.x：一个引擎解决 BM25 + kNN + 结构化过滤，架构最简

选型过程中做了 PoC 对比测试：分别用 3 种方案跑了 10 个代表性 query，对比召回结果的相关性和响应时间。ES 的混合搜索在 "精确店名 + 模糊语义" 这种混合场景下明显优于单一方案。

**为什么这是难的**：不是某个技术问题难，而是 **决策的不可逆性带来的压力**。选错了搜索引擎，后期迁移成本巨大。没有团队讨论、没有 senior 把关，需要自己对结果负责。应对方式是：**小规模 PoC 验证，用数据而不是直觉做决策**。

## Part 9: 反思与改进

### 如果重新做，会改变什么？

1. **数据层**：summary 是中文拼接的，对印尼语/英语用户搜索时 BM25 打折扣。重来会三语各生成一份 summary + embedding。实际已通过独立语言 system prompt 缓解（en-id 和 id-id 有专门的英文/印尼语 prompt，要求 LLM 翻译中文 summary 后回答）。
2. **前端架构**：初版用自定义正则解析 Markdown，后迁移到 react-markdown + 自定义组件映射，鲁棒性和可维护性大幅提升。后续可进一步考虑用 AI SDK 的 `streamObject` 输出结构化数据。
3. **测试覆盖**：已建立 48 场景意图测试 + 22 场景 E2E 测试，但缺少单元测试和回归测试。核心链路应有自动化 CI 集成测试。
4. **延迟优化**：TTFB ~2s 主要是 Gemini generateObject 的固有延迟。可以考虑简单查询用规则匹配跳过 LLM（如纯城市+菜系查询），只在复杂场景走 LLM。
