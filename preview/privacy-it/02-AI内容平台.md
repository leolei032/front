### Q1-1: 为什么选 Next.js 而不是前后端分离？

两个关键需求决定了必须用 SSR 框架：

1. **SEO 需求**：120w+ 多语言落地页的程序化 SEO，纯 SPA 对爬虫不友好
2. **首屏性能**：面向印尼市场，设备和网络条件参差不齐，SSR 让首屏更快

选 Next.js 而不是 Nuxt/Remix：团队 React 技术栈更熟悉；ISR 能力适合"内容更新频率低但页面量巨大"的场景；App Router 的 Server Components 天然减少客户端 JS bundle。

trade-off：API Routes 不适合重计算，复杂业务逻辑应该拆独立服务。

### Q1-2: 你说独立完成全流程，具体怎么理解"独立"？

独立是指技术侧的端到端交付。但不是完全没有协作：产品方向和业务目标跟 leader 对齐；UI 没有专职设计师，参考了 Airbnb/Booking/TripAdvisor 自己做；数据源是已有的业务数据。

### Q1-3: 为什么选 ES 而不是 PostgreSQL 全文搜索或 Pinecone？

| 维度 | PostgreSQL | Pinecone | Elasticsearch 8.x |
|------|-----------|----------|-------------------|
| BM25 全文搜索 | 支持 | 不支持 | 成熟 |
| kNN 向量搜索 | pg_vector 不够成熟 | 强 | 原生支持 |
| 结构化过滤 | 强 | 弱 | 强 |
| 中文分词 | 弱 | 不涉及 | smartcn 开箱即用 |

选 ES 因为一个引擎解决所有需求（BM25 + kNN + 结构化过滤），架构最简单。数据量 2400 条，单分片搞定。

### Q1-4: App Router vs Pages Router？遇到什么坑？

选 App Router 因为：Server Components 默认减少 JS bundle；嵌套布局更优雅；流式渲染配合 Suspense。

**遇到的坑**：
1. **Hydration Mismatch**：严格控制 `'use client'` 范围，数据获取和格式化留在 Server Components
2. **缓存行为不透明**：所有 fetch 显式设置 `cache` 和 `revalidate`，不依赖框架默认值
3. **Client Components 传染性**：`'use client'` 是边界线，import 的模块都降级为客户端代码。用 children/slot 模式嵌入 Server Component：

```tsx
// page.tsx (Server Component)
<ClientWrapper>
  <ServerChild />   {/* 仍然在服务端渲染，JS 不进 bundle */}
</ClientWrapper>
```

### Q1-5: shadcn/ui 的工作原理？和传统组件库的本质区别？

传统组件库是 npm 包（源码在 node_modules），shadcn/ui 是**代码生成器**——把组件源码复制到你的项目里。意味着：完全可控可改、没有版本依赖、按需引入、底层用 Radix UI 无障碍原语。

项目中 shadcn 组件放 `components/ui/`（禁止直接导入），上层封装 `components/common/qpon-ui/` 暴露给业务。

### Q2: AI 对话式搜索是怎么设计的

**回答（PCTCR）**：

**P**：用户想通过自然语言找餐厅，如"雅加达适合约会的日料"，传统搜索无法理解复合意图。

**C**：2400+ 篇文章维度复杂；需要支持多轮对话；首字节延迟控制在 600ms 内。

**T**：四阶段 RAG 架构，核心思路是"LLM 做意图理解和内容生成，检索和上下文管理交给确定性代码"：

```
用户 query
  -> Stage 1: Gemini 结构化提取（城市、菜系、价位、场景等）+ 意图路由
    -> Stage 2: Session 合并（代码逻辑，非 LLM）
      -> Stage 3: ES 混合检索（kNN + BM25 + 结构化过滤）
        -> Stage 4: Gemini 流式生成推荐（SSE 推送）
```

**C**：
- Stage 1 用 `generateObject`（Zod Schema 约束）确保城市 ID、菜系名在白名单内
- Stage 2 用代码做上下文继承：硬筛选有新值覆盖、无则继承；语义查询根据话题切换决定追加还是重置
- Stage 3 混合检索 ES DSL：

```json
{
  "query": { "bool": { "filter": [
    { "term": { "city": "jakarta" } },
    { "terms": { "cuisine": ["japanese"] } }
  ] } },
  "knn": {
    "field": "content_embedding",
    "query_vector": [0.12, -0.34, ...],
    "k": 20, "num_candidates": 100
  },
  "rank": { "rrf": {} }
}
```

- Stage 4 LLM 流式输出 Markdown + `[POI:articleId]` 标记，前端实时解析渲染为图文混排

**R**：首字节约 550ms，完整推荐 8 家餐厅约 3-5s。支持多轮对话。

### Q2-1: 为什么分两次 LLM 调用？合成一次不行吗？

分开是为了可控性。合成一次时 LLM 经常：提取的城市 ID 格式不对、忘记上一轮条件、检索参数和生成不一致。分开后：Stage 1 用 Schema 约束提取质量稳定；检索和上下文合并是确定性代码；Stage 4 只做一件事。代价是多 150ms，换来可控性值得。

### Q2-2: 多轮对话上下文怎么管理？

客户端维护 SessionContext，每轮随请求发送，服务端用代码合并：

```typescript
interface SessionContext {
  city: string | null;
  cuisine: string | null;
  lastSemanticQuery: string;
  seenArticleIds: string[];
  seenArticlesData: ArticleSummary[];
}
```

合并逻辑：硬筛选有新值覆盖、无则继承；菜系变了（话题切换）重置查询和已推荐列表。

**为什么不传历史给 LLM**：消耗大量 token 增加延迟；LLM 记忆是概率性的偶尔遗忘；代码合并是确定性的可调试。

### Q2-3: 意图路由四种类型为什么这么分？

不同意图处理链路完全不同：
- **new_search**：走完整 Stage 2->3->4
- **refinement**（"更便宜的？""对比前两家"）：不需要重新检索，用 Session 已推荐数据
- **follow_up**（"这家在哪？"）：定位目标 POI 直接回答
- **chitchat**（"你好"）：预设回复

refinement 和 follow_up 省掉检索环节，响应更快。

### Q2-4: 前端流式渲染怎么做？

1. 服务端先通过 SSE data-part 发送 POI 元数据（文章信息、封面图、评分），前端建立映射表
2. LLM streamText 流式输出 Markdown，插入 `[POI:articleId]` 标记
3. 前端正则实时解析，拆分为三种 ContentBlock：普通文本(react-markdown) / POI卡片 / 追问建议按钮

### Q2-5: 如何防止 LLM 幻觉？

三层防护：
1. **Prompt 约束**：明确要求只推荐 context 中的文章
2. **RAG 架构**：LLM 知识边界被限制在检索结果内
3. **后处理校验**：检查推荐的 articleId 是否在检索结果中

抽查 100 条推荐，幻觉率 < 2%。

### Q2-6: 为什么选 Gemini 而不是 GPT-4 / Claude？

| 维度 | Gemini 2.0 Flash | GPT-4 | Claude 3 |
|------|-------------------|-------|----------|
| 成本 | ~$3.5/M tokens | ~$30/M tokens | ~$15/M tokens |
| 延迟 | 首字 ~150ms | 首字 ~500ms | 首字 ~400ms |
| GCP 集成 | 原生 Vertex AI | 需额外配置 | 需额外配置 |

成本是 GPT-4 的 1/10；同机房调用延迟更低；预留了模型切换能力。单次搜索成本约 $0.01。

### Q3: P95 延迟 146ms、QPS 900+ 是怎么做到的

**四级缓存命中链路**：

```
用户请求
  -> PWA Service Worker Cache (短 TTL, 离线可用)
    -> CDN Edge Cache (就近节点, 中等 TTL)
      -> Next.js ISR Cache (stale-while-revalidate)
        -> Redis (7d TTL, API层兜底)
          -> ES / BigQuery (数据源)
```

绝大多数请求在前两层被拦截，穿透到数据源的极少。

### Q3-1: 缓存一致性怎么保证？

**最终一致性**策略（内容更新频率低）：
- **主动失效**：内容更新时触发 CDN purge API + Redis DEL + `revalidatePath`
- **被动过期**：各级 TTL 自然过期（PWA < CDN < ISR < Redis）
- **兜底机制**：stale-while-revalidate，用户始终有内容看

```typescript
async function invalidateArticle(slug: string) {
  revalidatePath(`/article/${slug}`);  // Next.js 缓存
  await redis.del(`article:detail:${slug}`);  // Redis
  await cdnPurge(`/article/${slug}`);  // CDN（可选）
}
```

### Q3-2: 缓存穿透/击穿/雪崩怎么防？

**穿透**（查不存在的数据）：空值缓存 5 分钟

```typescript
if (!result) {
  await redis.set(key, 'NULL', 'EX', 300);
  return null;
}
```

**击穿**（热点 key 过期瞬间）：ISR 的 stale-while-revalidate 天然避免（过期后仍返回旧缓存），所以未实现互斥锁。

**雪崩**（大量 key 同时过期）：TTL 加随机抖动

```typescript
const ttl = 3600 + Math.floor(Math.random() * 300);
```

### Q3-3: 压测数据

| 场景 | QPS | 成功率 | P50 | P95 | P99 |
|------|-----|--------|-----|-----|-----|
| 首页单压 | **903** | 100% | 22.9ms | 146.4ms | 188.0ms |
| 推荐单压 | **887** | 100% | 19.9ms | 180.0ms | 224.2ms |
| 组合压测 | **780** | 100% | 19.0ms | 183.7ms | 234.1ms |

50 并发、600 秒/场景、10 秒超时。**诚实说明**：GCP 内网压测，不含用户端网络延迟。

### Q3-4: 扩到 5000 怎么做？

- GKE HPA 水平扩展 Pod
- Redis 不是瓶颈（2400 条数据）
- ES 不是瓶颈（单分片 <50ms）
- 真正瓶颈在 LLM 调用：Gemini API 有 QPM 限制，需要请求排队或模型降级

## Part 4: 程序化 SEO（120w+ 页面）

### Q4: 120w+ 多语言落地页怎么实现

基于 Next.js 动态路由 + ISR 按需生成：
- **动态路由模板**：`[locale]/[category]/[city]/[slug]`
- **Sitemap 分片**：每个 sitemap 最多 50000 条 URL，按语言分片
- **结构化数据**：JSON-LD（Restaurant / Article / ItemList）
- **Hreflang**：每个页面声明三语 alternate，双向引用 + x-default
- **generateMetadata**：自动生成 title / description / OG 卡片 / Hreflang / Canonical

**ISR + SEO 策略**：热门 100 篇预生成（generateStaticParams），其余按需生成（fallback: 'blocking'）。

SEO 效果：Lighthouse SEO 92 分，上线 1 个月纯 SEO 流量 1200+ 用户。

## Part 6: 数据处理与 AI 元数据

### 追问链路

```
"2000+ 篇文章的 AI 元数据提炼，具体提炼了什么？"
-> "批量处理的流程？速率限制怎么处理？"
-> "向量化用什么模型？维度多少？"
```

### 回答

**离线处理两步**：

**Step 1 - 搜索数据提取**（Python 脚本）：对每个 POI，从 BigQuery 抓取 Google 评论（>250字，最多50条）和 Instagram 帖子（近6个月，最多20条），用 Gemini Flash 提取 8 个结构化字段：定位、菜系、价格、场景、招牌菜、口味、氛围、贴士。支持断点续跑和 5 并发。

**Step 2 - ES 写入**：菜系标准化映射、价格区间映射、设施信息提取，拼接 200-300 字 summary，调用 multilingual-e5-large 模型生成 1024 维向量，bulk 写入 ES。

**选 e5-large**：自托管免费（vs OpenAI ada-002 $0.1/M tokens）；三语支持好；自托管延迟 ~20ms（vs API ~200ms）。

**为什么不在线提取**：一个 POI 50 条评论提取需 3-5 秒；评论不频繁变化；离线可做质量检查。

## Part 7: GCP 部署与运维

### 追问链路

```
"Cloud Build + GKE 的部署流程是什么？"
-> "GKE 的 Pod 资源配置？HPA 策略？"
-> "出了线上问题怎么排查？回滚策略？"
```

### 回答

**部署流程**：代码推送 -> Cloud Build 触发 -> Docker 多阶段构建（standalone output）-> Artifact Registry -> GKE 滚动更新

**GKE 配置**：
- Deployment + Service + Ingress + HPA
- Pod 内 PM2 cluster mode 充分利用多核 CPU
- 环境变量通过 Secret Manager 管理

**多 Pod 缓存一致性**：Redis 是共享中心化缓存，所有 Pod 连同一个实例；ISR 缓存各 Pod 独立（冷启动时少量请求稍慢）；CDN 在前面兜底。

**回滚**：kubectl rollout undo 或 Cloud Build 重新部署指定版本。
