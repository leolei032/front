### 题 1：设计一个前端埋点 SDK（你有实战经验，优势题）

#### 核心架构

```
                    SDK Core
                   /    |    \
            采集层   处理层   上报层
           /  |  \    |      |   \
        自动  手动  错误  格式化  批量  离线
        采集  埋点  监控  +校验  上报  缓存
```

#### 关键 API

```typescript
// 初始化
tracker.init({
  appId: 'qpon',
  endpoint: '/api/track',
  sampling: 0.1,        // 10% 采样
  batchSize: 20,        // 批量上报
  flushInterval: 5000,  // 5s 定时上报
});

// 手动埋点（类型安全）
tracker.track('click', { element: 'buy_btn', page: 'detail' });

// 自动采集
tracker.enableAutoTrack(['pageview', 'click', 'exposure']);

// 用户标识
tracker.identify(userId);
```

#### 要考虑的问题

- 采样策略：全量 vs 抽样，用户级采样 vs 事件级采样
- 上报时机：批量 + 定时 + 页面卸载（sendBeacon）
- 离线缓存：IndexedDB 存储，网络恢复后重传
- 性能影响：不阻塞主线程，requestIdleCallback 执行
- 数据格式标准化：统一的事件 schema
- 防重复：事件去重 ID（timestamp + random）

### 题 2：设计一个 JSBridge

#### 核心架构

```
Web 端                          Native 端
  |                                |
  |-- JSBridge.call('api', data) ->|
  |   (URL Scheme / postMessage)   |
  |                                |-- 执行 Native 能力
  |<- callback(result) ------------|
  |   (eval / postMessage)         |
```

#### 关键 API

```typescript
// Web 端调用 Native
const result = await bridge.call('getLocation', { accuracy: 'high' });

// Native 调用 Web
bridge.on('pushNotification', (data) => {
  // 处理推送
});

// 带超时的调用
const result = await bridge.call('scanQR', {}, { timeout: 10000 });
```

#### 要考虑的问题

- 双向通信：Web -> Native（URL Scheme / prompt 劫持 / postMessage）和 Native -> Web（直接调 JS）
- 回调管理：callbackId 映射表，避免回调泄漏
- 超时处理：Promise + 超时自动 reject
- 版本兼容：能力检测 bridge.canIUse('getLocation')
- 权限控制：白名单 / 签名校验
- 调试工具：Bridge 调用日志，Mock 模式

### 题 3：设计一个前端性能监控平台

#### 核心架构

```
采集层                处理层              展示层
  |                    |                  |
Web Vitals SDK  ->  上报网关  ->  数据聚合  ->  Dashboard
自定义指标       |  数据清洗  |  P50/P75/P95  |  告警规则
错误监控         |  采样控制  |  维度分析      |  趋势对比
资源监控         |  反压保护  |  （机型/网络） |  CI 集成
```

#### 关键指标

```
核心指标：LCP / FID(INP) / CLS / FCP / TTFB
业务指标：首屏可交互时间 / 接口成功率 / 白屏率
资源指标：JS 加载耗时 / 图片加载耗时 / 长任务
```

#### 要考虑的问题

- 采集：PerformanceObserver API / web-vitals 库 / 自定义 mark/measure
- 上报：采样率控制 + sendBeacon + 批量 + 降级到 Image
- 聚合：分位值（P50/P75/P95）而不是平均值
- 维度：按机型、系统、网络、页面、版本切分
- 告警：基于基线的动态阈值，而不是固定阈值
- 和 CI 集成：Lighthouse CI 分数趋势

### 题 4：设计一个通用 Hooks 库

#### 分类体系

```
hooks/
  state/        - useToggle, useSet, useMap, useLocalStorage
  lifecycle/    - useMount, useUnmount, useUpdateEffect
  dom/          - useClickOutside, useHover, useScroll, useResize
  async/        - useRequest, useDebounce, useThrottle, usePolling
  sensor/       - useNetwork, useMediaQuery, useInViewport
```

#### 设计原则

- 单一职责：一个 hook 只做一件事
- 参数设计：必选参数 + options 对象（可选参数）
- 返回值：数组形式（便于重命名）或对象形式（字段多时）
- SSR 安全：window/document 访问都要做判断
- 测试：用 @testing-library/react-hooks

## Part D：回答原则

1. **PCTCR 结构回答项目问题**：Problem -> Constraint -> Thinking -> Crafting -> Result
2. **每个数字都要有来源**："3s -> 1.7s" 要说清怎么测的、测的什么
3. **主动暴露 trade-off**：完美答案反而可疑，主动说局限性
4. **不懂就说不懂**："这块我没深入研究过，但根据我的理解推测是..."
5. **区分"我做的"和"团队做的"**：归因准确是信任的基础
6. **面试官水平判断**：如果面试官在某个方向持续追问，说明这是他的强项，要谦虚回应
