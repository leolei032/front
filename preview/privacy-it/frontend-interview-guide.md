# 前端面试官的项目考察框架

> 目标：通过项目经历，判断候选人的**技术层次**和**真实能力边界**。

---

## 一、核心考察逻辑

面试官的所有问题本质上是在做**分层判定**：

| 层次 | 信号 | 典型表现 |
|------|------|----------|
| **初级** | 能做 | 复述功能和技术栈，"我用了 React + Redux" |
| **中级** | 知道为什么 | 讲清楚 trade-off，"选 Zustand 是因为项目状态简单，Redux 的 boilerplate 没有必要" |
| **高级** | 能应对变化 | 给出决策框架，"如果团队规模从 3 人变成 10 人，我会重新考虑，因为..." |

面试官会根据你的回答**动态调整追问深度**，而不是按清单走。你回答得越深，问题就越难，直到触碰到你的能力边界。

---

## 二、六个考察维度

### 维度 1：技术选型与决策能力

**考察目的**：你是"跟着教程选"还是"根据场景选"。

**典型问题**：

- 为什么选 Next.js 而不是纯 React SPA？
- 状态管理方案是怎么确定的？考虑过哪些备选？
- CSS 方案为什么选 Tailwind？有没有遇到它不好解决的场景？

**追问方向**：

- 如果项目从 C 端转向 B 端，你的选型会变吗？
- 这个选型有没有在后期给你带来过麻烦？

**面试官在听什么**：

- 你是否了解备选方案的优劣
- 你的决策是否有明确的判断依据（团队熟悉度、项目规模、性能需求）
- 你是否承认选型的局限性（完美答案反而可疑）

---

### 维度 2：核心难点与问题解决

**考察目的**：你解决问题的真实能力，这是面试中**权重最高**的部分。

**典型问题**：

- 项目中最有挑战的技术问题是什么？
- 遇到过线上事故吗？怎么排查的？
- 性能优化做了哪些？怎么量化的？

**致命追问链**（面试官会一层层剥）：

```
问题是什么 → 怎么发现的 → 怎么定位根因的 → 考虑了哪些方案
→ 为什么选这个方案 → 具体怎么实现的 → 效果怎么度量的
→ 有没有引入新问题
```

**面试官在听什么**：

- 你能不能还原**完整的问题解决链路**，而不是只有结论
- 细节是否经得起推敲（说"优化了 50%"就要能说出怎么测的）
- 是你做的，还是团队做的你在复述

> **关键提醒**：面试官能分辨"包装"和"真实经历"。一个你真正解决过的小问题，远比一个你背下来的大故事有说服力。

---

### 维度 3：架构设计与工程素养

**考察目的**：你写的代码是"能跑"还是"能维护"。

**典型问题**：

- 项目的目录结构是怎么组织的？为什么这样分？
- 通用组件和业务组件怎么划分的？
- 团队协作中如何保证代码一致性？（lint、规范、CR）
- 有没有做过组件库或工具库的抽象？怎么设计 API 的？

**追问方向**：

- 新人入职能多快上手你的项目？
- 如果要把某个模块给另一个团队用，需要改多少？

**面试官在听什么**：

- 模块边界是否清晰
- 抽象层次是否合理（过度抽象和没有抽象一样扣分）
- 是否有工程化意识（不只是写功能，还关注可维护性）

---

### 维度 4：业务理解与交付意识

**考察目的**：你是"接需求的工具人"还是"有业务 sense 的工程师"。

**典型问题**：

- 这个项目解决了什么业务问题？核心指标是什么？
- 你的技术方案是怎么和产品需求对齐的？
- 上线后效果如何？有数据支撑吗？

**追问方向**：

- 有没有你主动提出的技术驱动的业务优化？
- 有没有对不合理的需求 push back 过？怎么沟通的？

**面试官在听什么**：

- 你是否关心技术之外的东西（用户体验、业务指标）
- 你是否能用数据说话，而不只是"感觉变快了"
- 你和非技术角色的协作能力

---

### 维度 5：技术深度与原理理解

**考察目的**：从你用过的技术出发，探测你的理解深度。

**典型路径**（从项目细节切入，而非干问八股）：

```
"你项目里用了虚拟列表"
→ "滚动时怎么计算可见区域？"
→ "IntersectionObserver 的回调是宏任务还是微任务？"
→ "和 requestAnimationFrame 的执行顺序是什么？"
```

```
"你用了 SSR"
→ "hydration 过程中发生了什么？"
→ "hydration mismatch 你遇到过吗？怎么解决的？"
→ "React 的 Selective Hydration 了解吗？"
```

**面试官在听什么**：

- 你是"用过"还是"理解"
- 遇到不会的问题，你是瞎猜还是基于已有知识推导
- 你的知识体系是否有纵深，而不只是横向铺开

---

### 维度 6：反思与成长潜力

**考察目的**：你有没有自驱力和技术判断力。

**典型问题**：

- 如果重新做这个项目，你会改变什么？
- 项目中有没有你觉得做得不好、但因为各种原因没改的地方？
- 最近在关注什么新技术？有没有在项目中尝试过？

**面试官在听什么**：

- 你是否有自我批判意识（只说好的 = 没有反思）
- 你对新技术的态度是"跟风"还是"评估后选择性采纳"
- 你是否在持续成长

---

## 三、项目陈述模板

**弃用 STAR**，改用更紧凑的 **PCTCR** 结构：

| 要素 | 说明 | 时间占比 |
|------|------|----------|
| **P** - Problem | 什么问题/背景 | 10% |
| **C** - Constraint | 关键约束条件 | 10% |
| **T** - Thinking | 你的分析和方案对比 | 30% |
| **C** - Crafting | 具体实现细节 | 30% |
| **R** - Result | 量化结果 + 遗留问题 | 20% |

**示例**：

> **P**：商详页长列表滚动卡顿，用户反馈明显掉帧。
>
> **C**：数据量 1w+，每项高度不固定，且需要和顶部吸顶 Tab 联动。
>
> **T**：对比了 react-window 和 react-virtuoso，前者不支持动态高度，后者包体积大且和我们的吸顶逻辑冲突，最终决定自己实现。
>
> **C**：基于 IntersectionObserver 做可见区域检测，用 ResizeObserver 缓存实际渲染高度，滚动时通过二分查找定位起始索引。吸顶 Tab 的联动通过 ScrollSpy 模式同步。
>
> **R**：FPS 从 30 提升到 58，LCP 减少 1.2s。遗留问题是快速滚动时偶尔有空白闪烁，后续可以加预渲染缓冲区优化。

这个结构的好处：**30 秒能讲完一个点**，面试官可以在任何环节切入追问。

---

## 四、面试中的隐藏评分项

这些不会被明确考察，但全程都在被观察：

| 隐藏项 | 加分 | 减分 |
|--------|------|------|
| **表达能力** | 复杂问题讲得清楚简洁 | 绕来绕去抓不住重点 |
| **诚实度** | "这块我不太确定，但我猜测是..." | 不懂硬编，被追问后前后矛盾 |
| **技术热情** | 讲到细节时眼里有光 | 全程像在背稿 |
| **协作信号** | "我和后端讨论后决定..." | "这些都是我一个人做的" |
| **归因准确** | 清楚区分自己做的和团队做的 | 把团队成果说成自己的 |

---

## 五、准备策略

### 你需要准备的不是"答案"，而是"素材库"

1. **选 2-3 个项目**，每个项目准备 3-5 个可深挖的技术点
2. 每个技术点用 PCTCR 结构整理，确保能在 30s-1min 内讲清楚
3. 对每个点，**预演面试官可能的 3 层追问**，准备好应对
4. 对用到的核心技术，确保理解到原理层（至少比 API 深一层）

### 避坑清单

- 不要背诵式地讲项目，要像在和同事讨论技术方案一样自然
- 不要只讲"做了什么"，重点讲"为什么这么做"和"还能怎么做"
- 不要回避项目中的问题和妥协，坦诚的反思比完美的叙述更有说服力
- 不要用模糊的量化（"提升了很多"），要么给具体数字，要么别提
